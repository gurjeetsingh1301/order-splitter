# Assessment Answers

## 1. What was your approach (thought process) to tackling this project?

I broke the problem into layers and worked inside-out — utilities and data models first, then business logic, then the HTTP layer on top. This way each layer could be tested in isolation before the next depended on it.

**Step 1 — Understand the core problem.** The domain is straightforward: split a dollar amount across stocks by percentage, compute share quantities, and decide when to execute based on market hours. I identified the two non-trivial problems early: floating-point money arithmetic and timezone-aware scheduling.

**Step 2 — Nail the money math.** Naive rounding when splitting allocations can cause the child amounts to sum to slightly more or less than `totalAmount` (e.g., three stocks at 33%/33%/34% of $100 produce $33.33 + $33.33 + $33.34 = $99.99 with truncation, or overflow with rounding). I used a floor-then-remainder strategy: floor all allocations to 2 decimal places except the last stock, which absorbs whatever remains. This guarantees the sum is always exact.

For the allocation validation (must sum to 100%), I used integer-cents arithmetic: multiply each percentage by 100 and sum the integers, then compare to 10000. This avoids the `60 + 40 = 99.9999...` floating-point trap.

**Step 3 — Timezone-correct market hours.** The server might run in any timezone (UTC in production), so pinning decisions to "America/New_York" is critical. I used the built-in `Intl.DateTimeFormat` API with `{ timeZone: 'America/New_York' }` — zero dependencies, handles DST automatically.

**Step 4 — Keep the HTTP layer thin.** Controllers do nothing but parse the request and delegate to the service. All business logic lives in `order.service.ts`. This makes the service testable without spinning up HTTP.

**Step 5 — Test at every layer.** Unit tests for utilities (rounding, market hours), data layer (store), and service; integration tests via supertest for the full HTTP stack.

## 2. What assumptions did you make?

**Market hours:**
- US market hours are **9:30 AM – 4:00 PM ET, Monday–Friday**.
- **US market holidays are not accounted for.** Implementing this correctly requires a reliable holiday calendar (which changes year to year). The cost of a stale hardcoded list exceeds the benefit for this scope.
- **Pre-market (4:00–9:30 AM) and after-hours (4:00–8:00 PM) extended trading sessions are not supported.** The problem statement references standard market hours only.
- An order placed on **Friday after close schedules for Monday** (skipping the weekend), and likewise for Saturday and Sunday.
- A weekday order placed **before 9:30 AM is scheduled for today** — it will execute at market open, not the next day.

**Pricing:**
- The **default stock price is $100** when no `marketPrice` is provided in the request, as stated in the requirements.
- If a `marketPrice` is provided per stock, it overrides the default for that stock only.
- Market prices are **point-in-time** — the price in the request is what is used for quantity calculation. There is no real-time price feed.

**Data:**
- **All data is in-memory and does not persist across restarts**, as explicitly required.
- Order IDs are **UUIDs generated by `crypto.randomUUID()`** (built into Node.js ≥ 14.17, no dependency needed).

**API design:**
- `allocationPercent` values must be positive numbers and must **sum to exactly 100** across the portfolio.
- `orderType` is required on every order — there is no default.
- The `GET /orders` endpoint returns orders in insertion order (Map iteration order in V8 is stable).

## 3. What challenges did you face?

**Floating-point money arithmetic** was the most subtle challenge. The temptation is to write `totalAmount * allocationPercent / 100` and round, but this can produce sums that are off by a cent. The floor-then-remainder pattern solves it but requires care when there is only one stock (the single stock is the "last" stock and gets the full amount directly).

**Timezone handling without a library** required understanding `Intl.DateTimeFormat`'s `formatToParts` API, which is verbose but powerful. The tricky part was extracting the weekday as a named string (`"Mon"`, `"Tue"`, etc.) and mapping it to a numeric day-of-week — the `Date` object's `.getDay()` is always in local time, so it cannot be trusted here.

**Next-trading-day calculation** needs to handle: Friday after close → Monday (+3 days), Saturday → Monday (+2), Sunday → Monday (+1). Adding days in UTC (not local time) is important to avoid DST-boundary bugs.

**Test isolation** for the in-memory store was straightforward once I added a `.clear()` method and called it in `afterEach`. Without this, tests would leak state into each other and produce order-dependent failures.

## 4. If migrating to a production environment, what changes and security controls would you put in place?

**Persistence:**
- Replace the in-memory `Map` with a proper database (PostgreSQL for relational queries, or a time-series DB if order analytics matter). The store interface (`save`, `getById`, `getAll`, `clear`) is already abstracted — swapping the implementation requires changing one file.
- Add database migrations (e.g., with Flyway or node-pg-migrate).

**Authentication & Authorization:**
- Add API key or OAuth 2.0 (JWT bearer token) authentication on all endpoints.
- Scope permissions: a read-only key should not be able to create orders.
- Rate limiting per client key to prevent abuse (e.g., with `express-rate-limit` backed by Redis).

**Input validation & security:**
- Zod validation already rejects malformed input. In production, add a maximum portfolio size limit (e.g., no more than 100 stocks per order) to prevent resource exhaustion.
- Set security headers via `helmet` (X-Content-Type-Options, X-Frame-Options, HSTS, CSP).
- Enforce HTTPS — TLS termination at the load balancer, reject plain HTTP.

**Observability:**
- Replace `console.log` response timing with structured logging (e.g., `pino`) that emits JSON, making logs parseable by Datadog, CloudWatch, or Elastic.
- Add distributed tracing (OpenTelemetry) with trace IDs propagated through request headers.
- Expose a `/health` endpoint for load balancer health checks and a `/metrics` endpoint for Prometheus scraping.

**Reliability:**
- Run multiple instances behind a load balancer. The in-memory store must be replaced with a shared store first (Redis or a database) for this to work correctly.
- Add a circuit breaker around any external price feed calls.
- Graceful shutdown: drain in-flight requests before exiting (`process.on('SIGTERM', ...)`).

**Market data:**
- Integrate a real-time market price API (e.g., Polygon.io or Alpha Vantage) instead of accepting client-supplied prices or defaulting to $100.
- Integrate a market holiday calendar API to correctly skip holidays in execution scheduling.

**CI/CD:**
- Run `npm test` and `npm run build` in CI on every pull request.
- Block merges if test coverage drops below a threshold.
- Use environment-specific `.env` files injected by the secret manager (AWS Secrets Manager, HashiCorp Vault), never committed to source control.

## 5. How and where did you use LLMs?

I used AI as an accelerator

**Where I made the decisions, LLM wrote the code:**

- **Floating-point money arithmetic:** I identified upfront that naive rounding on allocation splits would break — the classic $33.33 × 3 = $99.99 problem. I specified the floor-then-remainder strategy and directed Claude to implement it that way. Left to defaults, LLM would have used `Math.round` everywhere.

- **Integer-cents arithmetic for validation:** I specified that the Zod `.refine()` for allocation sum must use `Math.round(percent * 100)` summed and compared to `10000` — not `sum === 100` — because floating-point makes `60 + 40` occasionally not equal `100`. Claude implemented it, but the decision to do it this way was mine.

- **Project structure and layering:** I defined the separation — thin controllers, service layer for all business logic, store abstracted behind an interface. This makes the service unit-testable without HTTP and the store swappable to a real DB later.

**Where LLM accelerated routine work:**

- Scaffolding boilerplate (package.json, tsconfig, middleware wiring, route registration).
- **Timezone handling:** I chose `Intl.DateTimeFormat` with `America/New_York` deliberately — zero dependencies, built into V8, handles DST automatically. I instructed Claude to use `formatToParts` to extract weekday/hour/minute in ET rather than relying on `Date.getDay()` which is local-time only. This is a subtle bug that catches people out in production (server in UTC, logic in ET).
- Writing the test cases once I defined what needed to be tested (boundary conditions for market hours, amount reconciliation assertions, store isolation via `afterEach`).
- Generating documentation drafts (README, API.md) which I then reviewed and corrected.

**In short:** I used the LLM to move fast on things I already knew how to build. The design choices — data model, edge case handling, algorithmic decisions — were mine. The LLM typed them out.
